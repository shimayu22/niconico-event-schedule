本ドキュメントは、**運用コスト・完全無料寄り**を優先し、**Google フォーム＋スプレッドシート＋静的フロント**へ寄せる流れで**ここまで決まったこと**と**未決定事項**をまとめたものです。

**リポジトリの現状**：**Next.js は撤去済み**（2026-04）。**サイト本体は `web/` に Astro で実装済み**（公開 CSV をビルド時に取得する静的生成）。

## 2. 確定していること（運用・仕様）

### 2.1 全体像

- **専用の X（Twitter）アカウント**を作成し、そこから **Google フォームへのリンク**等を案内する。
- **フォーム回答**の保存先は **Google スプレッドシート**（フォーム設定でリンク）。
- **フロントエンド**は **スプレッドシート上の「掲載してよいデータ」**のみを読んで表示する（公開URLは **掲載用**に限定）。
- **運用は作業者1人**。シート側の作業（掲載反映・削除）は **手動**から始める。応募が非常に増えたら **自動化・半自動化**を検討。
- **掲載前チェック**：原則 **手動で内容確認**してから掲載用シートに載せる（スパム・誤投稿対策）。
- **削除依頼**：**X（DM またはメンション）** 等、**受け付けチャネルを1つに固定**し、**掲載シートから行削除**する運用。最終判断は運営と利用規約で明文化を検討。

### 2.2 Google フォームで集めたい項目（現在想定）

| 項目 | メモ |
|------|------|
| 投稿者名（活動名・ハンドル等） | 本名想定ではない想定。公開前提。 |
| 投稿祭の名前 | |
| X アカウント | **必須ではなくてよい**（フォーム方式のため）。 |
| ニコニコ動画のURL | **必須**。リンク用＋**サムネ組み立て**に使用。 |
| 開始日 / 終了日 | |
| カテゴリ | **選択式** |

**スプレッドシート単体が（掲載用について）見られてもよい**判断で進めている（受付〜未掲載の扱いは運用で分離）。

#### 掲載用シート：列定義（決定）

フォーム回答が入る **受付シート**の列名は Google フォーム任せになりがちなので、**フロントが読むのは掲載用シートだけ**とする。**受付 → 掲載への転記**のときに、下表の **1行目（ヘッダー）** に合わせて貼り付ける／コピーする。

| 列名（1行目・ヘッダー） | §2.2 との対応 | 内容・形式 | 必須 |
|------------------------|---------------|------------|------|
| `id` | （運用・§2.2 外） | **UUID**（v4 想定。例: `xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx`）。共有 URL のアンカー・更新時の同一性用。Google スプレッドシートでは Apps Script の `Utilities.getUuid()` 等で生成し、**再計算で変わらないよう値貼り付けで固定**する運用を推奨。 | はい |
| `published_at` | （運用・§2.2 外） | **掲載用シートに行を載せた日時**（一覧の「掲載の新しい順」ソート用）。**JST** 基準で、**RFC 3339** 推奨: `YYYY-MM-DDTHH:mm:ss+09:00`（例: `2026-04-12T15:30:00+09:00`）。列内で形式を統一する。 | はい |
| `submitter_name` | 投稿者名 | 公開表示用。 | はい |
| `event_name` | 投稿祭の名前 | カードタイトル等に使う。 | はい |
| `x_handle` | X アカウント | シートでは **`@` なし**のハンドル文字列を入れる運用を推奨（表示時に `@` を付与してリンク化する想定）。`@` 付きで入ってもよい（フロントで正規化）。無い場合は**空欄**。 | いいえ |
| `niconico_url` | ニコニコ動画のURL | **完全な watch URL**（`https://www.nicovideo.jp/watch/sm…` 等）**または動画 ID のみ**（`sm12345678` / `nm…` 等）を許容。フロントでリンク・サムネ用に **正規化**する。 | はい |
| `start_date` | 開始日 | **JST の暦日**を `YYYY-MM-DD`（例: `2026-04-01`）。 | はい |
| `end_date` | 終了日 | 同上。**`start_date` ≤ `end_date`** とする。 | はい |
| `category` | カテゴリ | **ニコニコ動画のジャンル相当の日本語ラベル**を、そのまま入れる（許容値は `web/src/lib/categories.ts` と **完全一致**。未定義の文字列は行としては有効だが **カテゴリ別ページには出ない**）。 | はい |

- **空行**はフロントでは無視する想定。
- **ヘッダー行は上記の ASCII 名のみ**（スペースなし）に統一し、公開 CSV でもそのまま使えるようにする。
- **一覧の並び順**は原則 **`published_at` の降順**（新しく掲載した行が先頭）とする想定。`id`（UUID）はソートキーにしない。**開催終了後にトップから外す場合の扱い** は § 2.7。

**当面フォームを使わない運用**でも、上記ヘッダー・列の意味は変えず、**掲載用シートに直接入力**する（§2.6）。

### 2.3 フロント（`web/` の実装）

- **Astro**・出力は**静的 HTML**（ビルド成果物は `web/dist/`）。**Next.js は撤去済み**（2026-04）。
- **データ取得**：ビルド時に **`fetch`** で **公開 CSV**（掲載用シートの「ウェブに公開」URL）を読む。環境変数 **`PUBLIC_EVENTS_CSV_URL`** にその CSV の URL を設定する（ローカルは `web/.env`、CI はリポジトリ Secrets 等。 [README.md](../README.md) 参照）。
- **ページ**
  - **`/`（トップ）**：**開催終了前**のイベントのみカード一覧（§ 2.7）。並びは **`published_at` 降順**。
  - **`/ended/`（開催終了イベント）**：**開催終了済み**の行のみ。並びは **`end_date` 降順**、同一終了日なら **`published_at` 降順**（§ 2.7）。
  - **`/terms/`（利用規約）**：当面はプレースホルダ。本文の法務表現は後で整備。
  - **`/category/{slug}/`（掲載中・カテゴリ別）**・**`/ended/category/{slug}/`（終了・カテゴリ別）**：`category` 列と `web/src/lib/categories.ts` のラベルが一致する行。`slug` は ASCII（例: `music-sound`）。**定義済みジャンル 17 件は常に静的ページを生成**（件数 0 でも空表示。`astro dev` でセレクト先が 404 にならないようにする）。
- **カテゴリでの絞り込み**：トップ／終了一覧・月別ページでは **セレクト**でジャンル選択（ニコニコ動画のジャンル並びに準拠。`web/src/lib/categories.ts` を正とする）。ルート生成の前提は **`/category/…`** の項と同様（`PUBLIC_EVENTS_CSV_URL` が無いビルドでは空になり得る）。**月とカテゴリの併用 URL は未対応**（どちらか一方の一覧で絞り込み）。
- **ナビゲーション**：トップ・終了一覧・利用規約へのリンク。**投稿フォーム** は URL 確定後にナビへ載せる想定（現状は「準備中」表示）。
- **正規化**：`niconico_url`（watch 完全 URL または `sm` / `nm` 等の ID）、`x_handle`（`@` の有無）はフロントで統一（列定義は § 2.2）。
- **サムネイル**：`ext.nicovideo.jp/api/getthumbinfo` で `thumbnail_url` を取得し、`nicovideo.cdn.nimg.jp` のパスは表示品質のため **`.L`（360×270）** にそろえる。取得失敗時はプレースホルダ。**より高解像度・カード上の見た目** はデザイン詰めで検討（§ 3 も参照）。
- **開始月での絞り込み**：`start_date` の **年月（JST の暦月、`YYYY-MM`）** ごとに **`/month/YYYY-MM/`**（掲載中）・**`/ended/month/YYYY-MM/`**（終了）を用意。トップ／各一覧に月ナビを表示。**ビルド時** CSV から当該リストに行がある月のページを生成（`PUBLIC_EVENTS_CSV_URL` が無いビルドでは月別ルートは空）。

### 2.4 非機能・開発ルール（継続）

- **SEO は優先しない**（従来どおり）。
- **お金のかからない構成**を優先。
- **HTTP クライアント**：**axios は使わない**方針（`fetch` のみ。 [README.md](../README.md)）。

### 2.5 スプレッドシート：公開CSV vs Apps Script JSON

**段階的な方針（決定）**：

1. **当面（テスト〜UI 確定まで）**  
   **公開 CSV** を使う。本番の Google フォーム運用を始める**前**でもよく、**掲載用シートに手で必要列（§2.2）を入力**して試す。データ取得と一覧表示（デザイン）を固める。
2. **その後**  
   運用・非公開シートの分離などを踏まえ、**Apps Script 等で JSON を返す方式**へ**移行を検討**する。  
   **列スキーマ（フィールドの意味・ヘッダー名）は CSV 時代とそろえる**想定にすると、フロントは取得 URL とパースの差し替えで済みやすい。

比較のメモ：公開 CSV は設定が素直。JSON はシート本体を非公開にしやすいが、GAS デプロイ・CORS・エンドポイント URL の整理が増える。

### 2.6 当面の運用（フォーム未使用・掲載シート手入力）

- **しばらく Google フォームは使わない**。作業者が **掲載用シート**に、§ 2.2 の列定義どおり **直接行を追加・編集**する。
- **公開データ** は引き続き **掲載用シート由来の CSV** のみとし、フロントはそれだけを読む。
- フォーム受付を再開する場合は、§ 2.1・§ 2.2 の「受付 → 掲載への転記」に沿って運用を整理する。

### 2.7 開催終了の扱いと、静的ビルドにおける「今日」

**（決定）開催終了の定義**

- シートの **`end_date`** は **JST の暦日**（`YYYY-MM-DD`）。**`start_date` ≤ `end_date`**（§ 2.2）。
- JST の「今日」を `todayYmd`、`end_date` を `endYmd` とするとき、**`todayYmd > endYmd` なら開催終了** とする（**最終日 `endYmd` 当日は「掲載中」** としてトップに載せる）。

**ページ分け**

- **トップ `/`**：開催終了**でない** 行のみ表示する。
- **`/ended/`**：開催終了**のみ** を表示する。
- トップに載せる行が 0 件で終了分だけ存在する場合は、トップから **`/ended/` への案内** を出す（実装済み）。

**静的サイトで「今日」が更新されるタイミング**

- 上記の比較に使う **「今日」は、ビルド処理が走った時点の JST 暦日** である（クライアント側で日付が勝手に変わるわけではない）。
- **暦日が変わっただけでは一覧は自動では切り替わらない。** 反映させるには **再ビルド・再デプロイ** が必要。
- **再ビルド・再デプロイ**は、**`main` への push**、**定時**、**手動**などでトリガできる必要がある（シート更新だけでは日付は切り替わらない）。実装として **GitHub Actions**、**Cloudflare Pages の Git 連携**、またはその組み合わせを使う。**本番の配信先・リポジトリの公開可否は §2.8**。
- ビルド時に **`PUBLIC_EVENTS_CSV_URL`** を環境変数で渡す（ローカルは `web/.env`、CI / Pages のビルド設定は各ホスティング側。手順は [README.md](../README.md)）。

### 2.8 ホスティング（決定：Cloudflare Pages + 非公開 GitHub）

**決定（2026-04）**：

- **静的ホスティング**：**Cloudflare Pages** を使う。
- **GitHub リポジトリ**：**非公開（private）** とする。

**検討の経緯（要約）**：

| 論点 | 内容 |
|------|------|
| 無料・静的・独自ドメイン | **無料**で静的配信・**独自ドメイン**（例：`shimay.uno` の**サブドメイン**）を載せやすい。URL はサブドメインの方が、サブディレクトリ配下に載せるより構成が単純になりやすい（別途検討）。 |
| GitHub Pages との比較 | **GitHub Free** では GitHub Pages が**公開リポジトリ向け**の位置づけのため、このリポジトリのように **`doc/` に内部メモ・方針・日次メモを同居**させると、**リポジトリを公開したくない**要件と衝突する。**非公開リポジトリのまま**無料で Pages を使う手段は、プラン次第で別途有料が必要になり得る。 |
| Cloudflare Pages | Git 連携で **private リポジトリ**からビルドできる（**無料枠の範囲**で利用。詳細は [Cloudflare の Git 連携](https://developers.cloudflare.com/pages/get-started/git-integration/) など。 |
| 「公開」の意味 | **ビルド成果物（サイト）は URL にアクセスすれば閲覧できる**（静的配信の性質上、HTML/JS は取得可能）。**`doc/` などを Git で非公開に保ちたい**という目的には、**非公開 GitHub + Cloudflare Pages** が合う。 |
| ライセンス | リポジトリを**公開しない**場合でも LICENSE を置くかは任意。公開に切り替える際は `LICENSE` の検討を推奨。 |

**実装・手順（未了）**：

- 現状の **`.github/workflows/deploy-web.yml`** は **GitHub Pages 向け**の想定で書かれている。**Cloudflare Pages へ移行する際**は、**Pages の Git 連携**（push でビルド）に寄せるか、**Actions から Wrangler 等で Pages にデプロイ**するかを決め、ワークフロー・[README.md](../README.md) を更新する。
- **`PUBLIC_EVENTS_CSV_URL`** は Cloudflare Pages の **環境変数（ビルド時）** に設定する。
- **独自ドメイン**は Cloudflare（DNS 管理と Pages のカスタムドメイン）側で設定する。
