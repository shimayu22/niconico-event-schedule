## 1. 方針転換の要約

| 項目 | 旧（主に functional-spec / コード足場） | 新（本ドキュメントで整理） |
|------|------------------------------------------|----------------------------|
| フロント | Next.js（App Router） | **Astro**（静的寄り・後学のため採用検討） |
| データ | Supabase（Postgres + RLS + Storage） | **Google スプレッドシート**（掲載用をフロントが読む） |
| 投稿 | アプリ内フォーム＋Turnstile 等 | **Google フォーム**（リンクは **X 専用アカウント**等から案内） |
| 画像 | ユーザーが1枚アップロード | **必要に応じ廃止**。**ニコニコ動画URLからサムネ用URLを組み立て**て表示（取得不可時は**代替テキスト／プレースホルダ画像**） |
| モデレーション | RLS・管理者ログイン・行削除 | **作業者1人**。**受付シート→掲載シートへの手動反映**（応募増加時に自動化を検討） |

## 2. 確定していること（運用・仕様）

### 2.1 全体像

- **専用の X（Twitter）アカウント**を作成し、そこから **Google フォームへのリンク**等を案内する。
- **フォーム回答**の保存先は **Google スプレッドシート**（フォーム設定でリンク）。
- **フロントエンド**は **スプレッドシート上の「掲載してよいデータ」**のみを読んで表示する（公開URLは **掲載用**に限定）。
- **運用は作業者1人**。シート側の作業（掲載反映・削除）は **手動**から始める。応募が非常に増えたら **自動化・半自動化**を検討。
- **掲載前チェック**：原則 **手動で内容確認**してから掲載用シートに載せる（スパム・誤投稿対策）。
- **削除依頼**：**X（DM またはメンション）** 等、**受け付けチャネルを1つに固定**し、**掲載シートから行削除**する運用。最終判断は運営と利用規約で明文化を検討。

### 2.2 Google フォームで集めたい項目（現在想定）

| 項目 | メモ |
|------|------|
| 投稿者名（活動名・ハンドル等） | 本名想定ではない想定。公開前提。 |
| 投稿祭の名前 | |
| X アカウント | **必須ではなくてよい**（フォーム方式のため）。 |
| ニコニコ動画のURL | **必須**。リンク用＋**サムネ組み立て**に使用。 |
| 開始日 / 終了日 | |
| カテゴリ | **選択式** |

**スプレッドシート単体が（掲載用について）見られてもよい**判断で進めている（受付〜未掲載の扱いは運用で分離）。

### 2.3 フロント（計画）

- **Astro** を使う前提。**プロジェクトはリポジトリの `web/` に置く**（`doc/` とサイト本体の役割を分ける）。**Next.js は撤去済み**（2026-04）。
- **メイン画面**：イベント一覧を **カード表示**。
- **表示制御**：**カテゴリ**、**開催月**（ユーザー要件として言及済み）。
- その他：**フォームへのリンク**、**利用規約**ページ（またはセクション）。
- **サムネ**：ニコ動の仕様に沿って URL 組み立て。**取得失敗時は代替**（文言または画像）。

### 2.4 非機能・開発ルール（継続）

- **SEO は優先しない**（従来どおり）。
- **お金のかからない構成**を優先。
- **HTTP クライアント**：**axios は使わない**方針（`fetch` のみ。 [README.md](../README.md)）。

### 2.5 スプレッドシート：公開CSV vs Apps Script JSON

どちらを採用するかは **未確定**（比較は会話済み。手動運用・小規模なら **公開CSV** から試す選択肢あり）。

## 3. 未決定・要詰め

| 項目 | 状態 |
|------|------|
| **データ取得方式** | **公開CSV** か **Apps Script の JSON** か（CORS・キャッシュ・運用の好み）。 |
| **シート構成** | **受付のみ** / **掲載のみ** の2枚に分けるか、1枚＋列で **「掲載OK」** フラグを付けるか。 |
| **ニコニコサムネ URL の組み立て規則** | 動画 ID の切り出し・画像 URL 形式の**最新仕様確認**（仕様変更の可能性）。**アクセス集中時のリスク**の受け止め。 |
| **カテゴリの値一覧** | フォームの選択肢と**フロントのフィルタ**を一致させる定義。 |
| **開催月フィルタ** | JST 基準、**開始日を月に刺すか終了日も考慮するか** 等。 |
| **終了イベントの扱い** | 別ページか、トグルで隠すか（旧仕様では「終了は別ページ」— 静的サイトでも踏襲するか要確認）。 |
| **ホスティング** | GitHub Pages / Cloudflare Pages / Vercel（静的）等。 |
| **利用規約・問い合わせ文言** | 掲載条件・削除依頼・公開範囲の明記。 |
| **旧資産（`doc/old/supabase/migrations` 等）** | 採用しない方針なら **アーカイブ**／README に「未使用」と注記するか。
