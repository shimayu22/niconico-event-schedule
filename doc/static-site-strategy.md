## 2. 確定していること（運用・仕様）

### 2.1 全体像

- **専用の X（Twitter）アカウント**を作成し、そこから **Google フォームへのリンク**等を案内する。
- **フォーム回答**の保存先は **Google スプレッドシート**（フォーム設定でリンク）。
- **フロントエンド**は **スプレッドシート上の「掲載してよいデータ」**のみを読んで表示する（公開URLは **掲載用**に限定）。
- **運用は作業者1人**。シート側の作業（掲載反映・削除）は **手動**から始める。応募が非常に増えたら **自動化・半自動化**を検討。
- **掲載前チェック**：原則 **手動で内容確認**してから掲載用シートに載せる（スパム・誤投稿対策）。
- **削除依頼**：**X（DM またはメンション）** 等、**受け付けチャネルを1つに固定**し、**掲載シートから行削除**する運用。最終判断は運営と利用規約で明文化を検討。

### 2.2 Google フォームで集めたい項目（現在想定）

| 項目 | メモ |
|------|------|
| 投稿者名（活動名・ハンドル等） | 本名想定ではない想定。公開前提。 |
| 投稿祭の名前 | |
| X アカウント | **必須ではなくてよい**（フォーム方式のため）。 |
| ニコニコ動画のURL | **必須**。リンク用＋**サムネ組み立て**に使用。 |
| 開始日 / 終了日 | |
| カテゴリ | **選択式** |

**スプレッドシート単体が（掲載用について）見られてもよい**判断で進めている（受付〜未掲載の扱いは運用で分離）。

#### 掲載用シート：列定義（決定）

フォーム回答が入る **受付シート**の列名は Google フォーム任せになりがちなので、**フロントが読むのは掲載用シートだけ**とする。**受付 → 掲載への転記**のときに、下表の **1行目（ヘッダー）** に合わせて貼り付ける／コピーする。

| 列名（1行目・ヘッダー） | §2.2 との対応 | 内容・形式 | 必須 |
|------------------------|---------------|------------|------|
| `id` | （運用・§2.2 外） | **UUID**（v4 想定。例: `xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx`）。共有 URL のアンカー・更新時の同一性用。Google スプレッドシートでは Apps Script の `Utilities.getUuid()` 等で生成し、**再計算で変わらないよう値貼り付けで固定**する運用を推奨。 | はい |
| `published_at` | （運用・§2.2 外） | **掲載用シートに行を載せた日時**（一覧の「掲載の新しい順」ソート用）。**JST** 基準で、**RFC 3339** 推奨: `YYYY-MM-DDTHH:mm:ss+09:00`（例: `2026-04-12T15:30:00+09:00`）。列内で形式を統一する。 | はい |
| `submitter_name` | 投稿者名 | 公開表示用。 | はい |
| `event_name` | 投稿祭の名前 | カードタイトル等に使う。 | はい |
| `x_handle` | X アカウント | シートでは **`@` なし**のハンドル文字列を入れる運用を推奨（表示時に `@` を付与してリンク化する想定）。`@` 付きで入ってもよい（フロントで正規化）。無い場合は**空欄**。 | いいえ |
| `niconico_url` | ニコニコ動画のURL | **完全な watch URL**（`https://www.nicovideo.jp/watch/sm…` 等）**または動画 ID のみ**（`sm12345678` / `nm…` 等）を許容。フロントでリンク・サムネ用に **正規化**する。 | はい |
| `start_date` | 開始日 | **JST の暦日**を `YYYY-MM-DD`（例: `2026-04-01`）。 | はい |
| `end_date` | 終了日 | 同上。**`start_date` ≤ `end_date`** とする。 | はい |
| `category` | カテゴリ | **フォームの選択肢と同じキー**を入れる（表示ラベルはフロントでマッピング可）。**許容する値の一覧**は §3「カテゴリの値一覧」確定後にフォーム・本シート・フロントでそろえる。 | はい |

- **空行**はフロントでは無視する想定。
- **ヘッダー行は上記の ASCII 名のみ**（スペースなし）に統一し、公開 CSV でもそのまま使えるようにする。
- **一覧の並び順**は原則 **`published_at` の降順**（新しく掲載した行が先頭）とする想定。`id`（UUID）はソートキーにしない。

### 2.3 フロント（計画）

- **Astro** を使う前提。**プロジェクトはリポジトリの `web/` に置く**（`doc/` とサイト本体の役割を分ける）。**Next.js は撤去済み**（2026-04）。
- **メイン画面**：イベント一覧を **カード表示**。
- **表示制御**：**カテゴリ**、**開催月**（ユーザー要件として言及済み）。
- その他：**フォームへのリンク**、**利用規約**ページ（またはセクション）。
- **サムネ**：ニコ動の仕様に沿って URL 組み立て。**取得失敗時は代替**（文言または画像）。

### 2.4 非機能・開発ルール（継続）

- **SEO は優先しない**（従来どおり）。
- **お金のかからない構成**を優先。
- **HTTP クライアント**：**axios は使わない**方針（`fetch` のみ。 [README.md](../README.md)）。

### 2.5 スプレッドシート：公開CSV vs Apps Script JSON

**段階的な方針（決定）**：

1. **当面（テスト〜UI 確定まで）**  
   **公開 CSV** を使う。本番の Google フォーム運用を始める**前**でもよく、**掲載用シートに手で必要列（§2.2）を入力**して試す。データ取得と一覧表示（デザイン）を固める。
2. **その後**  
   運用・非公開シートの分離などを踏まえ、**Apps Script 等で JSON を返す方式**へ**移行を検討**する。  
   **列スキーマ（フィールドの意味・ヘッダー名）は CSV 時代とそろえる**想定にすると、フロントは取得 URL とパースの差し替えで済みやすい。

比較のメモ：公開 CSV は設定が素直。JSON はシート本体を非公開にしやすいが、GAS デプロイ・CORS・エンドポイント URL の整理が増える。

## 5. 次の実務ステップ（参考）

1. Google：**専用 X** → **フォーム** → **受付スプレッドシート**作成。
2. **掲載用**シートを用意し、**§2.2 内「掲載用シート：列定義」**どおりのヘッダーで運用開始（受付からの転記ルールは必要に応じて追記）。

4. **サムネ URL** ルールのプロトタイプ → 失敗時フォールバック。
