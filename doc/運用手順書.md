# 投稿祭スケジュールサイト — 運用手順書

本ドキュメントは、<strong>各自がホスティングする</strong>投稿祭スケジュールサイト（`web/` の Astro を **Cloudflare Pages** で公開）の、日常運用と自動デプロイの整理です。

## 1. 構成の前提

| 項目 | 内容 |
|------|------|
| サイト本体 | **自分の** Cloudflare Pages プロジェクト（GitHub の `main` と連携） |
| データ | ビルド時に `PUBLIC_EVENTS_CSV_URL` で指定した **自分の公開 CSV** を取得 |
| 開催区分 | **ビルド実行時点の JST の日付**で「開催中 / 開催予定 / 終了」を判定（日付が変わると一覧が変わり得る） |
| 日次ビルド（任意） | GitHub Actions が **毎日 15:01 UTC**（**日本時間 翌日 0:01**）に Deploy Hook を `POST` して再ビルド |
| 随時ビルド（任意） | Google Apps Script の `rebuildSite()` が同じ Deploy Hook を `POST`（シート更新直後の反映用） |

初回の本番構築（Pages のビルド設定、環境変数、Deploy Hook、GitHub Secret、DNS など）は [本番初回セットアップ手順.md](./本番初回セットアップ手順.md) を参照してください。ここでは**構築済み**を前提に日常運用のみを扱います。

---

## 2. スプレッドシートを更新するとき

1. **掲載用スプレッドシート**を編集する（列仕様・必須項目は [static-site-strategy.md](./static-site-strategy.md) §2.2）。
2. サイトが読む **CSV が、その内容を反映した状態で URL から取得できる**ようにする。
   - 例: スプレッドシートを **ファイル → 共有 → ウェブに公開** などで CSV 化し、その公開 URL を `PUBLIC_EVENTS_CSV_URL` に合わせておく。
   - **重要:** ビルドは「URL を fetch した瞬間」の内容を使います。**CSV 側の更新が完了してから** 再デプロイしてください。
3. （任意）編集内容を記録したい場合は、別途バージョン管理やバックアップの運用を行う。

`id` 列は UUID を生成するために、次の式を使えます。

```
=MAKE_UUID()
```

## 3. スプレッドシートを更新したあとにやること

**ゴール:** 自分の Cloudflare Pages で **新しい CSV を取り込んだビルド**を走らせる。

### 3.1 すぐサイトに反映したい場合（推奨フロー）

1. CSV が公開 URL 上で更新済みであることを確認する（ブラウザで URL を開く等）。
2. 次のいずれかで **再ビルド**を起動する。
   - **Google Apps Script**（設定済みの場合）
     - スクリプト プロパティに `DEPLOY_HOOK_URL` を設定済みであること。
     - `rebuildSite()` を実行する（エディタの実行、またはカスタムメニュー／トリガー）。
   - **GitHub Actions**（手動）
     - リポジトリの **Actions** → **Trigger Cloudflare Pages rebuild** → **Run workflow**。
   - **`main` への push**（コード変更がなくても空コミット等でトリガー可能だが、Deploy Hook の方が一般的）
3. **Cloudflare** → **Workers & Pages** → **自分のプロジェクト** → **Deployments** で、新しい行が **Triggered by deploy hook**（または Git 連携）付きで増え、<strong>Success</strong> になることを確認する。
4. 本番 URL（`*.pages.dev` やカスタムドメイン）を開き、表示を確認する。

### 3.2 すぐは叩かず、日次バッチに任せる場合

- **毎日 0:01 前後（JST）**の自動ビルドまで待てば、<strong>その時点の CSV</strong> で再ビルドされます（下記 §4）。
- 当日中に必ず反映したいイベントがある場合は、<strong>§3.1 の随時ビルド</strong>を使ってください。

### 3.3 コードやテンプレだけ変えた場合

- **`main` へ push** すると、Cloudflare Pages の **通常の Git 連携**でもビルドが走ります（Deploy Hook は不要）。
- データだけ変えて **リポジトリにコミットがない**ときは、<strong>Deploy Hook</strong>（§3.1）か **日次**（§4）に依存します。

## 4. 毎日 0:01（JST）前後の定期デプロイ（任意）

### 4.1 目的

- **日付が変わったあと**、同じ CSV でも「開催中 / 開催予定 / 終了」の表示が変わるため、<strong>毎日一度はビルド</strong>しておく。
- シートを触らない日でも、<strong>カレンダーに沿った一覧の正しさ</strong>を保つ。

### 4.2 仕組み

- リポジトリの **`.github/workflows/trigger-cloudflare-pages.yml`** が、cron `1 15 * * *`（UTC）で動作します。
- **15:01 UTC = 日本時間では翌日の 0:01** に相当するため、<strong>日付変更直後</strong>に近いタイミングでビルドが走ります。
- ワークフローは **Cloudflare の Deploy Hook URL** に `POST` するだけです（新しい git コミットは不要）。
- 実行には GitHub の **Repository secrets** に **`CLOUDFLARE_PAGES_DEPLOY_HOOK_URL`** が設定されている必要があります。

### 4.3 確認方法

- **GitHub** → **Actions** → **Trigger Cloudflare Pages rebuild** の実行履歴（スケジュール実行はタイムラグがある場合あり）。
- **Cloudflare** → **Deployments** で、適切な時刻前後の **Deploy hook** 由来のデプロイ。

### 4.4 注意

- GitHub の無料プランでは、<strong>スケジュール実行が少し遅れる</strong>ことがあります。**厳密な 0:01 必須**なら、別途常時 cron 等の検討が必要です。
- **同じ短時間に** `main` への push と Deploy Hook を重ねると、<strong>ビルドが二重</strong>に走ることがあります（通常は問題になりにくいが、ビルド回数は消費します）。

## 5. ローカル開発

```bash
cd web
cp .env.example .env
# PUBLIC_EVENTS_CSV_URL= に自分の CSV URL を設定
npm install
npm run dev
```

## 6. トラブル時のチェックリスト

| 現象 | 確認すること |
|------|----------------|
| サイトにイベントが出ない | Cloudflare の **環境変数** `PUBLIC_EVENTS_CSV_URL` が設定されているか。CSV URL を直接開いて中身があるか。 |
| シートを更新したのに古い | CSV が本当に更新されているか。その後 **Deploy Hook** または **push** で再ビルドしたか。 |
| 定期ビルドが動かない | GitHub Secret `CLOUDFLARE_PAGES_DEPLOY_HOOK_URL` の有無。Actions の失敗ログ。 |
| Apps Script でエラー | スクリプト プロパティ **`DEPLOY_HOOK_URL`**（名前と値）。初回の外部アクセス許可。 |
| デプロイは成功だが表示がおかしい | ビルド時点の JST 日付と、CSV の `start_date` / `end_date` の形式（`YYYY-MM-DD`）。 |

---

## 7. 秘密情報の取り扱い

- **Deploy Hook の URL** は、知られていると第三者が**あなたの Pages プロジェクトのビルドを起動**できるため、<strong>GitHub Secret</strong>・**Apps Script のスクリプト プロパティ**にだけ置き、リポジトリやスクリーンショットに載せない。
- 漏れた疑いがあるときは、Cloudflare 側で **Deploy Hook を削除して作り直し**、GitHub / Apps Script の値を更新する。
