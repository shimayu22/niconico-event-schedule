# 機能仕様

本ドキュメントは、**ニコニコ動画の「投稿祭」スケジュールを、公式統合サイトに依存せず共有する**ためのWebアプリについて、要求・要件を整理したものです。インフラ・技術は [system-architecture.md](./system-architecture.md) を参照し、仕様確定に応じて両方を更新します。

---

## 1. 背景と目的

### 1.1 背景（課題認識）

| 項目 | 内容 |
|------|------|
| ドメイン | ニコニコ動画では、提示テーマに沿った動画を参加者が投稿する**投稿祭**がある。 |
| 既存サイト | かつて [nicoevents.jp](https://www.nicoevents.jp/) が存在したが、ニコニコ動画側へ**統合**された。 |
| 統合後の課題 | 統合後の体験が必ずしも望ましくなく、**手軽にスケジュールだけを集約・閲覧**したいニーズがある。 |
| 「後援会」 | 統合後の公式導線では**後援会**が付く前提となっており、運用負荷や心理的なしんどさを感じる主催者・利用者がいる可能性がある。 |
| 本サイトの立ち位置 | **非公式の掲示板的な場**として、主催者がイベントを登録し、閲覧者が**ここを見れば開催予定が分かる**状態を目指す（公式・後援会フローとの連携を前提とはしない）。 |

### 1.2 目的（プロダクトとして達成したいこと）

| ID | 目的 |
|----|------|
| G-01 | **投稿祭（当面は車載動画系）**の情報を、必須項目そろえて**登録**できる。 |
| G-02 | **トップページで一覧**でき、開催前／開催中／終了が分かる（表示範囲は後述の要件どおり）。 |
| G-03 | **見た目はシンプル**で、長時間閲覧しても疲れにくい。 |
| G-04 | 個人運用を想定し、**継続的な運用コスト**（手間・金銭）を抑える。セキュリティ上削ってはいけない部分は除く。 |
| G-05 | **利用規約・使い方・問い合わせ**の導線を用意する。 |
| G-06 | **X の書き込み API 料金を避ける**ため、告知の**自動ポストは行わない**。代わり、投稿確定時に **Discord** へ通知し、**手動投稿用 X アカウント**でそのまま貼れる**文案**を渡す運用とする（詳細は **§1.5、FR-17**）。 |

### 1.3 スコープ（コンテンツ）

| 項目 | 方針 |
|------|------|
| 対象カテゴリ | **当面は「車載動画」関連の投稿祭のみ**を想定する。 |
| カテゴリの拡張 | 将来、複数カテゴリ・フィルタを戻す可能性はあるため、**データモデルやコード上の伸びしろ**は検討してもよいが、**初版の機能要件に含める必要はない**。 |

### 1.4 認証方針（確定）

| 項目 | 内容 |
|------|------|
| 採用 | **案 A**：**ログイン不要**。投稿者の **表示名**および **X（旧 Twitter）識別子**を**フォームで必須入力**する。 |
| 理由 | **X の開発者向け規約・料金が変動しやすい**ため、OAuth／ログイン連携は採用しない。 |
| 補足 | なりすまし抑止は **CAPTCHA**、**運用による削除**等に寄せる。 |

### 1.5 運用フロー（Discord 通知 → 手動 X 投稿）

| 手順 | 内容 |
|------|------|
| 1 | 利用者がサイト上で投稿を**確定**する（Supabase に保存）。 |
| 2 | サーバ（Next.js Route Handler 等）が **Discord Incoming Webhook** へ通知する（**FR-17**）。通知文に **X にそのままコピペしやすいテキスト**（タイトル、開催期間、**当該投稿を指す一覧上の URL（ディープリンク）**、告知 URL 等）を含める。**URL の形は FR-17・FR-18**。 |
| 3 | 管理者が Discord を確認し、**手動投稿専用の X アカウント**からポストする（深夜・多忙時は遅延しうる）。 |
| 4 | **問い合わせ**は、利用者に **当該 X アカウントへのメンション（@）** を案内する（**FR-02, FR-16**）。公開リプライになる点・DM 不可の場合は**利用規約／使い方**で注意を書く（**BR-06**）。 |

| 役割 | 説明 | 想定操作 |

## 4. 機能要件（FR）

### 4.1 投稿

| ID | 要件 |
|----|------|
| FR-01 | 利用者は**新規投稿**できる。投稿は **(1) 入力フォーム → (2) 確認画面 → (3) 確定** の手順とする（詳細は FR-02〜04 とあわせて定義）。ボット対策は [system-architecture.md](./system-architecture.md) の CAPTCHA 等と整合。 |
| FR-02 | **確認画面**では、入力内容の**要約（送信前の最終確認）**を表示する。また、**投稿確定後は利用者本人による編集・削除ができない旨**を表示する。さらに**問い合わせ先**として、**手動投稿・告知用の X（Twitter）アカウント**へ **@メンション**して連絡してもらう旨を表示する（**ハンドルは運用で確定**し、`NEXT_PUBLIC_OFFICIAL_X_HANDLE` 等でサイトに埋め込む想定）。 |
| FR-03 | 投稿の入力項目は、**いずれも必須**とする：**投稿祭タイトル**、**本文**、**開催開始日**、**開催終了日**、**告知用 URL**、**サムネイル画像（1枚）**、**投稿者表示名**、**投稿者の X（旧 Twitter）識別子**（いずれも**フォームで手入力**。正規化・文字数上限は **BR-01**）。 |
| FR-04 | **本文**は、**プレーンテキスト**とし、**改行が反映**されること。詳細説明はニコニコ等の告知先へ誘導する想定のため、リッチテキストや Markdown は**初版では必須としない**。 |
| FR-05 | **告知用 URL** は HTTP(S) の URL とし、想定はニコニコ動画の動画 URL だが**ブログ・自作サイト等も許容**する。形式検証は **BR-05**。 |
| FR-17 | 投稿が **Supabase 上へ正常に保存された後**、**Discord Incoming Webhook** へ通知する。通知内容に **X 投稿用のコピペ文案**を含める（例：タイトル、開催期間、**本サイト上で当該投稿を指す URL**、告知 URL、投稿者表示名・X 識別子。体裁は実装時にテンプレート化）。**当該 URL** は **投稿専用の詳細ページではなく**、**一覧ページ上のディープリンク**とする（**FR-18**。例：アクティブ一覧で `?post={投稿ID}`。ページネーションと併用する場合は `?page=n&post={投稿ID}` 等。開いたときに**該当カードへスクロールまたはハイライト**する実装を想定）。**Webhook URL はサーバ側の環境変数**にのみ保持し公開しない。**Discord 送信に失敗**しても**利用者の投稿成功は取り消さない**（ログ記録し、可能なら再試行は実装裁量）。 |

### 4.2 一覧・トップ

| ID | 要件 |
|----|------|
| FR-06 | **サイトのトップページは「投稿一覧」**とする。 |
| FR-07 | **デフォルト表示**では、**開催終了日が現在以降（まだ終了していない）**投稿のみを一覧する。**開催開始日の降順**で並べる。 |
| FR-08 | **開催が終了した**投稿は、**トップのデフォルト一覧には含めない**。終了済み一覧は **別ページ（または別ルート）** で閲覧できること。 |
| FR-09 | 各投稿について、**開催前／開催中／開催終了**のいずれかが利用者に分かる表示（ラベル・バッジ等）を行う。判定基準は **BR-03**。 |
| FR-10 | **初版では、一覧のソート順を利用者が切り替える機能は提供しない**。並びは **FR-07** に固定する。 |
| FR-18 | **一覧の UI**：各イベントは**カード状**で表示する。**画面遷移による一覧の切り替えはページネーション**とする（無限スクロール必須としない）。**投稿1件につき専用の詳細ページ（例：`/events/[id]`）は初版では設けない**。投稿には **DB 上の ID** があり、**FR-17** のリンク用には一覧 URL に **投稿 ID をクエリで付与**する（例：`/?post={uuid}`。終了一覧を別ルートにする場合も同様に `?post=` を付けられること。ページ番号と併用する場合のパラメータ名は実装時に統一）。 |

### 4.3 編集・削除

| ID | 要件 |
|----|------|
| FR-11 | **投稿成立後の編集機能は、初版では提供しない**。 |
| FR-12 | **管理者**は、**いずれの投稿も削除**できる。 |
| FR-13 | **投稿者本人による削除・編集は初版では提供しない**（案 A と整合）。 |

### 4.4 情報ページ・問い合わせ

| ID | 要件 |
|----|------|
| FR-14 | **利用規約**を閲覧できるページ（またはセクション）を用意する。 |
| FR-15 | **使い方**（投稿の流れ、禁止事項の要約など）を閲覧できるページを用意する。 |
| FR-16 | **問い合わせ**について、**主な手段**として **§1.5 の手動投稿用 X アカウントへの @メンション**を**使い方・フッター等に明記**する。サイト内の**フォーム経由**の問い合わせは**必須としない**；必要になった場合は **§10.1** の補完案を検討する。 |

## 5. ビジネスルール・バリデーション（BR）

| ID | ルール |
|----|--------|
| BR-01 | **投稿者表示名**：フォームで必須。前後空白除去。文字数上限は実装時に定める。<br>**投稿者 X 識別子**：フォームで必須。**@`handle` 形式推奨**だが、実装時に前後空白除去・先頭 `@` の有無の正規化を定める。文字数上限は実装時に定める。 |
| BR-02 | **サムネイル画像**：ちょうど **1枚・必須**。**最大ファイルサイズ 5MB**（1280×720 前後・約1.6MB想定に余裕）。**許可形式**：少なくとも **JPEG / PNG / WebP**（実装でMIMEと拡張子を照合）。上限超過は**エラーで拒否**。 |
| BR-03 | **開催期間の粒度**：開始・終了は**日付のみ**（時刻は持たない）。**1日の区切り**（どのタイムゾーンで「今日」を判定するか）は **JST** を既定として実装時にコードと表示で統一する。 |
| BR-04 | **開催期間の妥当性**：<br>・**開始日 ＞ 終了日**のときは**エラー**とし、送信不可。<br>・**開始日が「今日」より前**の場合は**送信は許可**するが、**警告**を表示する（表現は UI で明示）。<br>・**極端に長い期間**の場合は**送信は許可**しつつ**警告**を表示する。**閾値（例：何日超）**は実装時に定数として定め、運用で調整可能とする。 |
| BR-05 | **告知 URL**：**http://** および **https://** を許可する。構文として妥当な URL でない場合は**エラー**。 |
| BR-06 | **問い合わせを X のメンションに依頼する**場合、**公開リプライ**になりうること、**誰でも内容が見える**こと、相手のアカウント設定で **DM 不可**のときはメンションのみとなることを、**利用規約または使い方**に簡潔に記載する。スパムメンション対策は **X 側のブロック等に寄せる**（初版ではサイト側に通報 UI は不要）。 |

## 6. 非機能要件（NFR）

| ID | 要件 |
|----|------|
| NFR-01 | 投稿・一覧の**体感速度**に配慮する（数値SLOは未設定）。 |
| NFR-02 | **ボット等の悪用**を抑止する（CAPTCHA ほか。詳細はアーキテクチャ与件）。 |
| NFR-03 | **不正アクセス**（他人の投稿の改ざん・無権限削除等）を許さない。RLS・認証設計と整合。 |
| NFR-04 | スパム・悪意ある投稿への**追加対策**（モデレーション、通報、承認制等）は、運用負荷とトレードオフのうえ**別途**。 |
| NFR-05 | **運用・コスト**は可能な限り低く（ホスティング無料枠、マネージドサービス。セキュリティ必要経費は除く）。 |
| NFR-06 | 個人でも**継続保守が破綻しにくい**構成・ドキュメントを志向する。 |
| NFR-07 | **UI はシンプル**にし、情報量・コントラスト・密度のバランスを取り、**長時間見ても疲れにくい**ことを重視する。 |
| NFR-08 | **SEO は優先しない**（既存方針）。 |

## 9. 対象外（初版）

- X / OAuth によるログイン・識別子の自動取得。
- **X API を用いた告知の自動ポスト**（**G-06**。手動投稿と **FR-17 の Discord 通知**で代替）。
- 投稿の編集、投稿者本人による削除。
- 一覧のソート切替。
- 公式ニコニコ・後援会システムとの自動連携。
- 複数カテゴリの切替 UI（データ準備のみ将来向けに残すのは可）。
- コメント・いいね・通報機能（要否は将来）。
- 多言語化。
- **投稿ごとの詳細専用ページ**（例：`/events/[id]`）。一覧は**カード＋ページネーション**と **`?post=` ディープリンク**に留める（**FR-18**）。

## 10. 未決定・運用で埋める項目

### 10.1 問い合わせ（FR-16）— 確定方針と補完案

**確定（運用方針）**

- **主要な問い合わせ手段**：**手動投稿・告知用に用いる X アカウント**へ、利用者に **@メンション**してもらう（**FR-02、FR-16、BR-06**）。
- **実装**：サイト上に **@`ハンドル`** を表示（例：`NEXT_PUBLIC_OFFICIAL_X_HANDLE`）。**ハンドル文字列の確定**が必要。

**補完案（必要になった場合のみ）**

- **非公開**の連絡や **X を使わない**利用者向けに、後から **サイト内フォーム＋Turnstile＋メール転送**（Resend 等）を追加する余地はある（以前案の A/B/C）。初版では**必須としない**。

| 候補 | 用途 |
|------|------|
| フォーム＋Turnstile＋転送 | メールを公開したくない・内容を非公開にしたい問い合わせ用の**オプション** |
| mailto: のみ | **非推奨**（スパム・ボットに弱い） |
