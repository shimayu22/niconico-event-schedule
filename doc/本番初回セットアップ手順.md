# 本番初回セットアップ手順（Cloudflare / GitHub / スプレッドシート / ムームードメイン）

このドキュメントは、**同じ構成を一から再現する**ためのチェックリストです。**日常のシート更新や再デプロイ**は [運用手順書.md](./運用手順書.md) を参照してください。

**機密情報**（Deploy Hook の URL、GitHub のトークン類、スプレッドシートの非公開 URL など）は**リポジトリに書かない**でください。手順では **設定画面の項目名**と**リポジトリ内で既に決まっている名前**（環境変数名・Secret 名）だけを記します。

---

## 全体の流れ（推奨順）

1. **Google スプレッドシート**で **掲載用シート** を用意し、**公開 CSV の URL** を取得する。
2. **Cloudflare**：Pages プロジェクトを作成し、**GitHub（非公開）と連携**、**ビルド設定**と**環境変数**、必要なら**カスタムドメイン**と **Deploy Hook** を設定する。
3. **GitHub**：Cloudflare 連携用の権限承認、**Actions 用 Secret**（Deploy Hook URL）を登録する。
4. **ムームードメイン**：ドメインの **DNS（レコード／ネームサーバー）** を、Cloudflare の指示どおりに合わせる。

順序の理由：**CSV URL** は Pages の **`PUBLIC_EVENTS_CSV_URL`** に必要。**Deploy Hook** は Pages 側で作成した URL を GitHub Secret に入れるため、Cloudflare を先に進めると迷いが少ないです。

## 1. Google スプレッドシート

### 1.1 掲載用シート

- **列定義・ヘッダー名・必須項目**は [static-site-strategy.md](./static-site-strategy.md) の「掲載用シート：列定義」を正とする。
- フロントは **公開 CSV だけ**を読む。受付用シートを分ける場合は、**掲載用**を CSV 化する。

### 1.2 CSV を URL で公開する

1. スプレッドシートを開く。
2. **ファイル → 共有 → ウェブに公開**（UI は Google の改版で表記が変わることがある）。
3. 公開対象を**掲載用シート**、形式を **CSV** にし、公開して **URL をコピー**する。
4. その URL を、後述の **`PUBLIC_EVENTS_CSV_URL`** に設定する。

**注意：** 公開 CSV は**インターネット上でも取得できる**前提です。載せてよいデータだけを掲載用に置く。

### 1.3 （任意）シート更新直後にサイトを再ビルドしたい場合

- Cloudflare の **Deploy Hook** を用意したうえで、**Google Apps Script** から同じ URL に `POST` する運用が取れる（詳細は [運用手順書.md](./運用手順書.md) の「随時ビルド」）。
- Apps Script 側では **スクリプト プロパティ**（例：プロパティ名 `DEPLOY_HOOK_URL`）にフック URL を保存し、リポジトリやスクリプト本文に直書きしない。

### 1.4 （任意）`id` 列の UUID

列定義上 `id` は UUID。スプレッドシート上で **カスタム関数**（例：`MAKE_UUID()`）や **一度生成したら値貼り付けで固定**する運用は [運用手順書.md](./運用手順書.md) も参照。

## 2. Cloudflare（Pages）

### 2.1 アカウント・ゾーン

- **Cloudflare** にログインし、**Workers & Pages** から Pages を利用する。
- **カスタムドメイン**を載せる場合、多くの構成で**当該ドメイン**を **Cloudflare に追加（ゾーン）**し、DNS を Cloudflare 側で管理する。**ムームードメインで取得したドメイン**でも、**ネームサーバー**を Cloudflare 指定に差し替える**パターンが一般的（後述）。

### 2.2 Pages プロジェクトの新規作成（Git 連携）

1. **Create → Pages → Connect to Git**。
2. **GitHub** を選び、リポジトリへのアクセスを許可（**非公開リポジトリ**にチェックが入ることを確認）。
3. 対象リポジトリ・**本番ブランチ（例：`main`）** を選ぶ。

### 2.3 ビルド設定（このリポジトリの前提）

Astro 本体は **`web/`** 以下にあるため、Pages の **Build** 設定は次を前提にする。

| 項目 | 設定例 |
|------|--------|
| Root directory（プロジェクトのルート） | `web` |
| Build command | `npm run build` |
| Build output directory | `dist` |

**Node.js バージョン：** `web/package.json` の `engines.node`（例：`>=22.12.0`）に合わせ、Pages の **Environment variables** で **`NODE_VERSION`**（例：`22`）を指定するか、Cloudflare が推奨する指定方法に従う。**ビルドログで Node が古くて失敗**する場合はここを疑う。

### 2.4 環境変数（ビルド時）

| 名前 | 説明 |
|------|------|
| `PUBLIC_EVENTS_CSV_URL` | **1.2** で得た**掲載用 CSV の公開 URL**（引用符は不要）。 |

**Production**（および Preview でもビルドを試すなら同様）に設定する。

### 2.5 デプロイの確認

- **Deployments** で **`main` の push** に応じたビルドが **Success** になることを確認する。
- 付与された **`*.pages.dev`** の URL で HTML が返ることを確認する。

### 2.6 Deploy Hook（日次・手動・Apps Script 用）

1. Pages プロジェクト → **Settings → Builds**（または Builds & deployments 配下）の **Deploy hooks**。
2. **新規フック**を作成し、**ブランチは `main`**（本番と同じ）を指定。
3. 表示された **URL をコピー**する（**秘密**として扱う）。

この URL は **GitHub の Secret** `CLOUDFLARE_PAGES_DEPLOY_HOOK_URL` と、必要なら **Apps Script のスクリプト プロパティ**にだけ保存する。ワークフロー例：`.github/workflows/trigger-cloudflare-pages.yml`。

### 2.7 カスタムドメイン

1. Pages プロジェクト → **Custom domains** → ドメインを追加（例：`schedule.example.com`）。
2. Cloudflare の画面上で、**DNS レコード**（多くは **CNAME**）または **SSL/TLS** の状態を **アクティブ**まで持っていく。
3. ドメインの **DNS を Cloudflare が握っている**構成なら、ウィザードで自動追加されることが多い。**外部 DNS のみ**の場合は、表示された **ターゲットホスト名**をムームー側に手入力する（次章）。

## 3. GitHub

### 3.1 リポジトリ

- **Private** を前提にする（方針は [static-site-strategy.md](./static-site-strategy.md) のホスティングを参照）。

### 3.2 Cloudflare 連携

- Pages 作成時に **GitHub App のインストール**と **リポジトリ選択**を済ませる。
- うまく連携しない場合：**GitHub → Settings → Applications** で Cloudflare の権限・対象リポジトリを確認する。

### 3.3 Actions 用 Secret（Deploy Hook）

1. リポジトリ → **Settings → Secrets and variables → Actions**。
2. **New repository secret**
   - **Name:** `CLOUDFLARE_PAGES_DEPLOY_HOOK_URL`
   - **Secret:** Cloudflare の Deploy Hook の **完全な URL**

3. **Actions** タブで **Trigger Cloudflare Pages rebuild**（`.github/workflows/trigger-cloudflare-pages.yml`）を **Run workflow** し、**成功**することを確認する。

### 3.4 スケジュール実行

- 同じワークフローは **cron `1 15 * * *`（UTC）** で **日本時間 翌日 0:01 前後**に相当する時刻で動く。GitHub の無料枠では **数分〜十数分の遅延**があり得る（運用の詳細は [運用手順書.md](./運用手順書.md)）。

## 4. ムームードメイン（DNS）

ムームードメインの管理画面名称は改版があり得るため、**「DNS」「ネームサーバー」**のメニューを探す。

### 4.1 推奨：ネームサーバーを Cloudflare に向ける

**Cloudflare にゾーンを追加したあと**、Cloudflare が提示する **ネームサーバー（NS）2 件**を、ムームードメイン側の **ネームサーバー設定** に登録する。

- 反映に **数時間〜最大 48 時間**かかることがある。
- 反映後、**WHOIS や dig** で NS が Cloudflare になっているか確認できる。
- メリット：**Pages のカスタムドメイン**と **SSL**、**DNS レコード**を Cloudflare 上で一元管理しやすい。

### 4.2 代替：ムームードメインで DNS を維持する

- Cloudflare Pages の **カスタムドメイン設定画面**に表示される **CNAME のターゲット**（例：`xxxx.pages.dev` など、画面の指示をそのままコピー）を、ムームー側の **DNS レコード**に追加する。
- **プロキシ（オレンジ雲）**は Cloudflare 側 DNS の話なので、**ムームー DNS のみ**の場合は **DNS のみの CNAME** となる（Pages の案内に従う）。

**注意：** ドメインが **Cloudflare に未追加**のままだと、カスタムドメインのウィザードで詰まることがある。手順は **Pages の画面の現在の文言**を優先する。

---

## 5. 完了条件（受け入れチェック）

- **`*.pages.dev`**（および設定した **カスタムドメイン**）でトップが表示される。
- **`PUBLIC_EVENTS_CSV_URL`** の CSV を更新し、**Deploy Hook** または **`main` への push** のあと、一覧が更新される。
- **GitHub Actions** の **Trigger Cloudflare Pages rebuild** が **手動で成功**する。可能なら **翌日の定時実行**も Deployments で確認する。
