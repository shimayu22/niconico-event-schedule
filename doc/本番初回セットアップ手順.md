# 本番初回セットアップ手順（Cloudflare Pages / GitHub / スプレッドシート）

**自分用の投稿祭スケジュールサイト**を **Cloudflare Pages** に初めてデプロイするためのチェックリストです。  
日常のシート更新や再デプロイは [運用手順書.md](./運用手順書.md) を参照してください。

**機密情報**（Deploy Hook の URL、GitHub のトークン類、非公開 CSV URL など）は**リポジトリに書かない**でください。手順では **設定画面の項目名**と**リポジトリ内で決まっている名前**（環境変数名・Secret 名）だけを記します。

---

## 全体の流れ（推奨順）

1. **Google スプレッドシート**で **掲載用シート** を用意し、<strong>公開 CSV の URL</strong> を取得する。
2. **Cloudflare**：Pages プロジェクトを作成し、<strong>GitHub と連携</strong>、<strong>ビルド設定</strong>と**環境変数**、必要なら**カスタムドメイン**と **Deploy Hook** を設定する。
3. **GitHub**：Cloudflare 連携用の権限承認、<strong>Actions 用 Secret</strong>（Deploy Hook URL・任意）を登録する。
4. **（任意）ドメイン登録業者**：カスタムドメインを使う場合、Cloudflare の指示どおり **DNS** を設定する。

順序の理由：**CSV URL** は Pages の **`PUBLIC_EVENTS_CSV_URL`** に必要。**Deploy Hook** は Pages 側で作成した URL を GitHub Secret に入れるため、Cloudflare を先に進めると迷いが少ないです。

---

## 1. 前提

| 項目 | 内容 |
|------|------|
| リポジトリ | GitHub **public**（本リポジトリを Fork しても、upstream をそのまま連携しても可） |
| ローカル | Node.js **22.12.0 以上**（`web/package.json` の `engines` 参照） |
| データ | 各自の **掲載用スプレッドシート**（列定義は [static-site-strategy.md](./static-site-strategy.md) §2.2） |
| Cloudflare | **自分のアカウント**で Pages プロジェクトを作成（他者のデプロイには影響しない） |

## 2. Google スプレッドシート

### 2.1 掲載用シート

- **列定義・ヘッダー名・必須項目**は [static-site-strategy.md](./static-site-strategy.md) の「掲載用シート：列定義」を正とする。
- フロントは **公開 CSV だけ**を読む。受付用シートを分ける場合は、<strong>掲載用</strong>を CSV 化する。

### 2.2 CSV を URL で公開する

1. スプレッドシートを開く。
2. **ファイル → 共有 → ウェブに公開**（UI は Google の改版で表記が変わることがある）。
3. 公開対象を**掲載用シート**、形式を **CSV** にし、公開して **URL をコピー**する。
4. その URL を、後述の **`PUBLIC_EVENTS_CSV_URL`** に設定する。

**注意：** 公開 CSV は**インターネット上でも取得できる**前提です。載せてよいデータだけを掲載用に置く。

### 2.3 （任意）シート更新直後にサイトを再ビルドしたい場合

- Cloudflare の **Deploy Hook** を用意したうえで、<strong>Google Apps Script</strong> から同じ URL に `POST` する運用が取れる（詳細は [運用手順書.md](./運用手順書.md) の「随時ビルド」）。
- Apps Script 側では **スクリプト プロパティ**（例：プロパティ名 `DEPLOY_HOOK_URL`）にフック URL を保存し、リポジトリやスクリプト本文に直書きしない。

### 2.4 （任意）`id` 列の UUID

列定義上 `id` は UUID。スプレッドシート上で **カスタム関数**（例：`MAKE_UUID()`）や **一度生成したら値貼り付けで固定**する運用は [運用手順書.md](./運用手順書.md) も参照。

## 3. Cloudflare（Pages）

### 3.1 アカウント

- **Cloudflare** にログインし、<strong>Workers & Pages</strong> から Pages を利用する。
- **カスタムドメイン**を載せる場合、多くの構成で**当該ドメイン**を **Cloudflare に追加（ゾーン）**し、DNS を Cloudflare 側で管理する（任意）。

### 3.2 Pages プロジェクトの新規作成（Git 連携）

1. **Create → Pages → Connect to Git**。
2. **GitHub** を選び、**Fork したリポジトリ**（または本リポジトリ）へのアクセスを許可する。
3. 対象リポジトリ・**本番ブランチ（例：`main`）** を選ぶ。

### 3.3 ビルド設定（このリポジトリの前提）

Astro 本体は **`web/`** 以下にあるため、Pages の **Build** 設定は次を前提にする。

| 項目 | 設定例 |
|------|--------|
| Root directory（プロジェクトのルート） | `web` |
| Build command | `npm run build` |
| Build output directory | `dist` |

**Node.js バージョン：** `web/package.json` の `engines.node`（例：`>=22.12.0`）に合わせ、Pages の **Environment variables** で **`NODE_VERSION`**（例：`22`）を指定するか、Cloudflare が推奨する指定方法に従う。**ビルドログで Node が古くて失敗**する場合はここを疑う。

### 3.4 環境変数（ビルド時）

| 名前 | 説明 |
|------|------|
| `PUBLIC_EVENTS_CSV_URL` | **2.2** で得た**掲載用 CSV の公開 URL**（引用符は不要）。**必須。** |
| `PUBLIC_CF_WEB_ANALYTICS_TOKEN` | Cloudflare Web Analytics の token（任意。未設定なら計測スクリプトは出さない）。 |

**Production**（および Preview でもビルドを試すなら同様）に設定する。

### 3.5 デプロイの確認

- **Deployments** で **`main` の push** に応じたビルドが **Success** になることを確認する。
- 付与された **`*.pages.dev`** の URL で HTML が返ることを確認する。

### 3.6 Deploy Hook（日次・手動・Apps Script 用・任意）

暦日が変わると「開催中 / 開催予定 / 終了」の表示が変わるため、**毎日一度は再ビルド**することを推奨します（[運用手順書.md](./運用手順書.md) §4）。

1. Pages プロジェクト → **Settings → Builds**（または Builds & deployments 配下）の **Deploy hooks**。
2. **新規フック**を作成し、<strong>ブランチは `main`</strong>（本番と同じ）を指定。
3. 表示された **URL をコピー**する（**秘密**として扱う）。

この URL は **GitHub の Secret** `CLOUDFLARE_PAGES_DEPLOY_HOOK_URL` と、必要なら **Apps Script のスクリプト プロパティ**にだけ保存する。ワークフロー例：`.github/workflows/trigger-cloudflare-pages.yml`。

Deploy Hook を使わない場合は、**シート更新のたびに `main` へ push** するか、Cloudflare ダッシュボードから手動デプロイでも構いません。

### 3.7 カスタムドメイン（任意）

1. Pages プロジェクト → **Custom domains** → ドメインを追加（例：`schedule.example.com`）。
2. Cloudflare の画面上で、<strong>DNS レコード</strong>（多くは **CNAME**）または **SSL/TLS** の状態を **アクティブ**まで持っていく。
3. ドメインの **DNS を Cloudflare が握っている**構成なら、ウィザードで自動追加されることが多い。**外部 DNS のみ**の場合は、表示された **ターゲットホスト名**をドメイン登録業者側に手入力する（次章）。

## 4. GitHub

### 4.1 リポジトリ

- **public** または **private** のどちらでも Cloudflare Pages と連携できます。Fork して各自運用する場合は **public** が一般的です。
- 本リポジトリを Fork した場合、**自分の Fork** を Pages に接続してください（upstream のデプロイ設定とは独立します）。

### 4.2 Cloudflare 連携

- Pages 作成時に **GitHub App のインストール**と **リポジトリ選択**を済ませる。
- うまく連携しない場合：**GitHub → Settings → Applications** で Cloudflare の権限・対象リポジトリを確認する。

### 4.3 Actions 用 Secret（Deploy Hook・任意）

日次・手動の再ビルド（`.github/workflows/trigger-cloudflare-pages.yml`）を使う場合:

1. リポジトリ → **Settings → Secrets and variables → Actions**。
2. **New repository secret**
   - **Name:** `CLOUDFLARE_PAGES_DEPLOY_HOOK_URL`
   - **Secret:** Cloudflare の Deploy Hook の **完全な URL**

3. **Actions** タブで **Trigger Cloudflare Pages rebuild** を **Run workflow** し、<strong>成功</strong>することを確認する。

### 4.4 スケジュール実行

- 同じワークフローは **cron `1 15 * * *`（UTC）** で **日本時間 翌日 0:01 前後**に相当する時刻で動く。GitHub の無料枠では **数分〜十数分の遅延**があり得る（運用の詳細は [運用手順書.md](./運用手順書.md)）。

## 5. カスタムドメインと DNS（任意）

ドメイン登録業者（ムームードメイン、お名前.com 等）の管理画面名称は改版があり得るため、<strong>「DNS」「ネームサーバー」</strong>のメニューを探してください。

### 5.1 推奨：ネームサーバーを Cloudflare に向ける

**Cloudflare にゾーンを追加したあと**、Cloudflare が提示する **ネームサーバー（NS）2 件**を、ドメイン登録業者側の **ネームサーバー設定** に登録する。

- 反映に **数時間〜最大 48 時間**かかることがある。
- 反映後、<strong>WHOIS や dig</strong> で NS が Cloudflare になっているか確認できる。
- メリット：**Pages のカスタムドメイン**と **SSL**、<strong>DNS レコード</strong>を Cloudflare 上で一元管理しやすい。

### 5.2 代替：登録業者側で DNS を維持する

- Cloudflare Pages の **カスタムドメイン設定画面**に表示される **CNAME のターゲット**（例：`xxxx.pages.dev` など、画面の指示をそのままコピー）を、登録業者側の **DNS レコード**に追加する。
- **プロキシ（オレンジ雲）**は Cloudflare 側 DNS の話なので、<strong>外部 DNS のみ</strong>の場合は **DNS のみの CNAME** となる（Pages の案内に従う）。

**注意：** ドメインが **Cloudflare に未追加**のままだと、カスタムドメインのウィザードで詰まることがある。手順は **Pages の画面の現在の文言**を優先する。

---

## 6. 完了条件（受け入れチェック）

- **`*.pages.dev`**（および設定した **カスタムドメイン**）でトップが表示される。
- **`PUBLIC_EVENTS_CSV_URL`** の CSV を更新し、<strong>Deploy Hook</strong> または **`main` への push** のあと、一覧が更新される。
- Deploy Hook を設定した場合、**GitHub Actions** の **Trigger Cloudflare Pages rebuild** が **手動で成功**する。可能なら **翌日の定時実行**も Deployments で確認する。
